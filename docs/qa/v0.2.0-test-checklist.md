# v0.2.0 Test Checklist

## 1. 目的

本清单用于固化 `v0.2.0` 在发布前必须完成的测试范围、执行方式和退出准则。

## 2. 自动化检查

发布前必须全部通过：

- `cargo fmt --all`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo test --workspace`
- `bash scripts/smoke_v0_2_0.sh`
- `git diff --check`

受限环境补充验证：

- `bash scripts/smoke_v0_2_0_sandbox.sh`

## 3. 核心功能验证范围

- 任务 `intake -> approve -> execute -> complete`
- 驳回后 `resubmit -> approve`
- 审批接口会拒绝与 `required_role` 不匹配的角色
- `context_reads` 与 `evidence` 输出
- `follow_up_items / follow_up_summary` 在 `tasks/intake` 与 `GET /api/v1/tasks/{task_id}` 中稳定输出
- `shift handoff` 请求会返回非空 `follow_up_items` draft 与非零 `follow_up_summary`
- `alert triage` 请求会返回非空 `follow_up_items` draft 与非零 `follow_up_summary`
- `POST /api/v1/tasks/{task_id}/follow-up-items/{follow_up_id}/accept-owner` 可把 seeded `follow_up_item` 从 `draft` 推进到 `accepted`
- `follow-up-items/{follow_up_id}/accept-owner` 会拒绝错误角色与已接受的 item
- `GET /api/v1/follow-up-items` 会跨任务返回最小 follow-up owner queue
- `GET /api/v1/follow-up-items` 支持 `owner_id / source_kind` 等过滤并保持默认优先级排序
- `GET /api/v1/follow-up-items` 支持 `blocked_only / escalation_required / due_before / risk / priority` 等运营 triage 过滤
- `GET /api/v1/follow-up-monitoring` 会返回最小 follow-up SLA monitoring 聚合视图
- `GET /api/v1/follow-up-monitoring` 支持与 `GET /api/v1/follow-up-items` 一致的过滤语义
- `handoff_receipt / handoff_receipt_summary` 在 `tasks/intake` 与 `GET /api/v1/tasks/{task_id}` 中稳定输出
- `shift handoff` 请求会返回非空 `handoff_receipt` draft 与非零 `handoff_receipt_summary`
- `POST /api/v1/tasks/{task_id}/handoff-receipt/acknowledge` 可把 `shift handoff` receipt 推进到 `acknowledged / acknowledged_with_exceptions`
- `handoff-receipt/acknowledge` 会拒绝与 `receiving_role` 不匹配的角色
- `POST /api/v1/tasks/{task_id}/handoff-receipt/escalate` 可把 `acknowledged_with_exceptions` 的 `shift handoff` receipt 推进到 `escalated`
- `handoff-receipt/escalate` 会拒绝错误状态和不匹配发送侧 accountable 角色的 actor
- `GET /api/v1/handoff-receipts` 会跨任务返回最小 cross-shift receipt queue
- `GET /api/v1/handoff-receipts` 支持 `shift_id / receipt_status / receiving_role / receiving_actor_id / overdue_only / has_exceptions / escalated_only` 过滤
- `GET /api/v1/handoff-receipt-monitoring` 会返回最小 handoff receipt monitoring 聚合视图
- `GET /api/v1/handoff-receipt-monitoring` 支持与 `GET /api/v1/handoff-receipts` 一致的过滤语义
- `alert_cluster_drafts / alert_triage_summary` 在 `tasks/intake` 与 `GET /api/v1/tasks/{task_id}` 中稳定输出
- `alert triage` 请求会返回非空 `alert_cluster_drafts` draft 与非零 `alert_triage_summary`
- `alert_cluster_drafts` 会稳定推断 `source_system / line_id / triage_label / recommended_owner_role / window`
- `scada` 阈值类 alert triage 请求会生成 `sustained_threshold_review` cluster draft 并路由到 `maintenance_engineer`
- `GET /api/v1/alert-clusters` 会跨任务返回最小 alert cluster queue
- `GET /api/v1/alert-clusters` 支持 `cluster_status / source_system / equipment_id / line_id / severity_band / triage_label / follow_up_owner_id / unaccepted_follow_up_only / follow_up_escalation_required / escalation_candidate / window_from / window_to / open_only` 过滤
- `GET /api/v1/alert-clusters` 会在每个 cluster item 上返回 `linked_follow_up` 摘要，覆盖 total/open/accepted/unaccepted/accepted_owner_ids/worst_effective_sla_status
- `GET /api/v1/alert-cluster-monitoring` 会返回最小 alert cluster monitoring 聚合视图
- `GET /api/v1/alert-cluster-monitoring` 支持与 `GET /api/v1/alert-clusters` 一致的过滤语义，包括 linked follow-up triage 过滤
- `GET /api/v1/alert-cluster-monitoring` 会返回 `linked_follow_up_clusters / unlinked_follow_up_clusters / accepted_follow_up_clusters / unaccepted_follow_up_clusters / follow_up_escalation_clusters`
- `GET /api/v1/alert-cluster-monitoring` 会返回 `follow_up_coverage_counts / follow_up_sla_status_counts` 两组 linked follow-up 聚合 bucket
- `planned_task.task.plan.governance` 输出
- `GET /api/v1/tasks/{task_id}`
- `GET /api/v1/tasks/{task_id}/evidence`
- `GET /api/v1/tasks/{task_id}/governance`
- `GET /api/v1/tasks/{task_id}/audit-events`
- `GET /api/v1/follow-up-items`
- `GET /api/v1/follow-up-monitoring`
- `GET /api/v1/handoff-receipts`
- `GET /api/v1/handoff-receipt-monitoring`
- `GET /api/v1/alert-clusters`
- `GET /api/v1/alert-cluster-monitoring`
- `GET /api/v1/audit/events` 过滤查询
- 文件模式重启后的任务、evidence、审计回读

## 4. 非功能检查

- 默认端口保持 `8000`
- `FA_SERVER_ADDR` 可覆盖本地监听地址
- `FA_DATA_DIR` 可稳定支撑重启回读
- SQLite / 文件 / 内存三种模式仍保持兼容
- 旧任务 JSON 缺少 follow-up task detail 字段时仍可兼容回读
- 旧任务 JSON 缺少 handoff receipt task detail 字段时仍可兼容回读
- 旧任务 JSON 缺少 alert cluster task detail 字段时仍可兼容回读
- 审批链和 evidence 输出在高风险任务下成立
- 高风险审批的角色约束不会被绕过

## 5. 手工验证补充点

- README 中的关键 API 示例可执行
- workflow 输出对现场角色具备基本可解释性
- 审计与 evidence 输出能支持审批理解
- 回退到人工流程的边界表述清楚

## 6. 退出准则

只有以下条件同时满足，`v0.2.0` 才可进入 `Ready for Release`：

1. 自动化检查全部通过。
2. 手工验证结果已记录。
3. 已知缺陷没有阻断主 workflow。
4. 变更、风险、回退方案已同步到文档。
