# M1 Pilot Workflow Specification

## 1. Workflow 名称

`设备温度漂移异常诊断与维修工单建议`

## 2. 目的

当关键设备出现重复的温度漂移异常时，平台需要帮助现场角色更快完成：

- 异常受理
- 上下文读取
- 诊断建议生成
- 维修建议审批
- 后续执行跟踪

目标不是替代人做设备控制，而是让异常处理从“经验驱动、口头协同”转向“证据驱动、可审计、可回放”的协同流程。

## 3. 触发条件

以下任一条件成立时，可触发该 workflow：

- 班组长发现设备温度连续超出正常波动阈值
- MES 发现同一设备重复出现温度相关异常记录
- 设备工程师主动发起诊断任务

## 4. 参与角色

| 角色 | 职责 |
| --- | --- |
| Production Supervisor | 发起任务，补充现场症状与业务影响 |
| Maintenance Engineer | 审阅诊断建议，决定维护动作 |
| Quality Engineer | 当异常可能影响质量时参与审批或升级 |
| FA Agent / Orchestrator | 读取上下文、生成建议、维持任务状态与审计 |
| System Admin / Delivery Owner | 维护运行边界、回退策略和试运行配置 |

## 5. 涉及系统与资产

| 对象 | 当前阶段作用 |
| --- | --- |
| MES | 提供生产上下文、异常记录、设备状态摘要 |
| CMMS | 提供历史工单、维护记录、维护建议上下文 |
| 设备本体 | 仅作为被诊断对象，不允许直接控制 |
| FA Platform | 编排任务、审批、审计、回放 |

## 6. 目标输出

workflow 的输出应包括：

- 异常摘要
- 证据清单
- 诊断建议
- 待审批维修建议
- 任务状态与审计轨迹

## 7. 流程步骤

### Step 1. 发起任务

发起人通过 `POST /api/v1/tasks/intake` 提交任务，请求中包含：

- 设备标识
- 异常描述
- 风险级别
- 期望结果

### Step 2. 读取上下文

编排器读取：

- MES 异常上下文
- CMMS 历史维修上下文

并把读取动作写入审计。

### Step 3. 生成计划与建议

平台基于风险等级和上下文生成执行计划，输出：

- 推荐的 agentic pattern
- 需要的人类参与角色
- 初步诊断路径
- 建议下一步动作

### Step 4. 审批门控

当建议动作涉及：

- 维修介入
- 停机检查
- 质量影响排查

则必须进入人工审批。

### Step 5. 执行跟踪

审批通过后，任务进入执行态，由人工或外部系统完成真实动作。平台只记录：

- 执行开始
- 执行完成或失败
- 关键说明

### Step 6. 审计回放

任务结束后，平台应能通过：

- `/api/v1/tasks/{task_id}`
- `/api/v1/tasks/{task_id}/audit-events`
- `/api/v1/audit/events`

回放整条处理轨迹。

## 8. Agentic Pattern 映射

| 阶段 | Pattern | 原因 |
| --- | --- | --- |
| intake / planning | `coordinator` | 需要汇总多角色与多系统信息 |
| evidence gathering | `ReAct loop` | 需要根据异常线索逐步形成诊断建议 |
| approval | `human-in-the-loop` | 维修建议和潜在停机动作必须由人工负责 |
| external action | `deterministic workflow + custom business logic` | 后续真实工单或设备动作不能直接交给模型 |

## 9. 证据与审计要求

必须记录：

- 任务发起人
- 任务 `correlation_id`
- 每次 connector 读取
- 审批请求与审批决定
- 执行开始、完成、失败

当前阶段允许的证据源：

- mock `MES`
- mock `CMMS`
- 人工补充说明

当前阶段证据输出方式：

- `POST /api/v1/tasks/intake` 返回 `evidence`
- `GET /api/v1/tasks/{task_id}` 返回 `evidence`
- `GET /api/v1/tasks/{task_id}/evidence` 返回任务级 evidence 快照列表

## 10. 明确禁止的动作

当前阶段明确不允许：

- 自动修改设备参数
- 自动下发 PLC / CNC / 机器人控制指令
- 自动创建真实 CMMS 工单
- 自动做质量放行或隔离决策

## 11. 回退策略

若 workflow 在任何节点出现异常，应回退到人工流程：

1. 记录失败原因
2. 保留任务与审计轨迹
3. 由现场负责人接手处理
4. 必要时新建后续任务继续跟踪

## 12. 与当前平台能力对齐

当前已支持：

- 任务 intake
- 任务查询
- 任务级 evidence snapshot
- 任务级 evidence 查询接口
- 审批
- 修订重提
- 执行开始
- 完成 / 失败
- 审计过滤与任务回放
- 文件 / SQLite / 内存三种本地运行模式

当前尚未支持但后续需要补齐：

- workflow 级可视化视图
- 与真实 CMMS 的受控写入草稿接口
- 角色责任矩阵的更强策略表达

## 13. 试运行验收标准

进入 `M2` 试运行前，至少满足：

1. 能完成 1 次端到端异常任务闭环演示。
2. 所有关键节点具备审计记录。
3. 审批路径和回退路径明确。
4. 现场角色能看懂输入、建议和输出责任边界。
5. 不发生越权自动执行。
