# M1 Alert Triage Candidate Workflow Specification

## 1. Workflow 名称

`产线告警聚合与异常分诊`

## 2. 文档定位

本文件不是当前 `v0.2.0` 已打通的正式 pilot workflow，而是高频、强时效、适合事件驱动扩展的日常协同 workflow specification baseline。

它的作用是：

- 把平台从人工任务 intake 扩展到更接近实时事件输入
- 明确告警场景下的聚合、去重、分级、路由和升级边界
- 为后续 `SCADA / Andon / MES / incident log` 输入建设提供设计基线

## 3. 目的

当产线出现高频告警、重复告警或短时间内多源告警集中爆发时，平台需要帮助现场角色更快完成：

- 告警聚合与去重
- 关键告警识别与优先级排序
- 初步分诊与责任路由建议
- 是否升级为正式异常任务的判断建议
- 风险提醒与审计回放

目标不是让 AI 自动做停线、消警或安全裁决，而是把“告警太多、现场靠经验筛选、升级标准不一致”的状态推进到“证据清晰、分诊一致、升级有边界、过程可追踪”的协同流程。

## 4. 触发条件

以下任一条件成立时，可触发该 workflow：

- 短时间窗口内出现告警风暴或重复告警
- 同一设备、线体或工位连续出现相似异常信号
- 班组长或值班主管主动请求告警分诊支持
- Andon / SCADA / MES 同时报告需要快速确认优先级的事件

## 5. 参与角色

| 角色 | 职责 |
| --- | --- |
| Production Supervisor | 确认现场业务影响、决定是否立即升级处理 |
| Shift Lead / Team Lead | 补充现场症状、产线状态和资源约束 |
| Maintenance Engineer | 对设备类告警提供初步解释和跟进建议 |
| Quality Engineer | 对可能影响质量的告警提供会签意见或升级建议 |
| Safety Officer | 对涉及安全风险的告警承担正式升级和处置责任 |
| FA Agent / Orchestrator | 聚合告警、提取证据、生成分级和路由建议、保留审计轨迹 |
| Delivery Owner / System Admin | 维护分诊模板、策略边界和运行开关 |

## 6. 涉及系统与资产

| 对象 | 当前阶段作用 |
| --- | --- |
| MES | 提供线体、工单、设备、产量和基础异常上下文 |
| SCADA / Andon | 提供实时告警、信号源和触发时间线 |
| Incident Log | 提供现场登记、处理备注和未关闭异常记录 |
| CMMS | 提供历史设备异常和未关闭维护事项上下文 |
| FA Platform | 编排任务、告警聚合、证据、路由建议、审计与回放 |

## 7. 目标输出

workflow 的输出应包括：

- 告警摘要
- 去重后的告警簇或告警组
- 建议优先级与分诊标签
- 建议 owner / 建议下一步
- 需要升级为正式异常任务的候选项
- 风险提醒与审计轨迹

## 8. 流程步骤

### Step 1. 发起告警分诊任务

发起人通过任务 intake 或受控事件入口提交告警分诊请求，输入至少包含：

- 告警时间窗口
- 线体 / 设备 / 区域范围
- 已知高风险信号
- 是否已有现场处理动作
- 期望输出对象，例如班组长、维修或值班主管

### Step 2. 读取告警与现场上下文

编排器读取：

- SCADA / Andon 告警记录与原始时间线
- MES 设备、工单和当前生产上下文
- incident log 中的现场备注或已知异常
- 必要时读取 CMMS 中历史异常或未关闭事项

并把所有读取动作写入审计。

### Step 3. 生成聚合、去重和分诊建议

平台基于上下文生成：

- 告警聚合结果
- 相似告警去重或归并结果
- 告警优先级排序
- 初步路由建议，例如生产、维修、质量或安全
- 需要升级为正式异常任务的候选项

### Step 4. 人工确认与必要升级

告警分诊默认是 assistive / coordination 输出，不直接执行真实业务动作。

若平台识别到以下情况，应提醒人工确认或升级：

- 告警涉及停线风险
- 告警涉及安全或人身风险
- 告警可能扩展为质量异常
- 告警持续时间或频率超过正常阈值

当前阶段的目标治理方向是：

- 普通告警分诊本身不要求正式审批
- 由 `Production Supervisor` 或等价现场负责人确认分诊结果
- 任何涉及安全、高风险停线、质量升级的真实动作，仍应回到已有 governed workflow 主链

### Step 5. 执行跟踪

分诊完成后，平台只记录：

- 哪些告警被归并为同一异常簇
- 哪些告警被标记为高优先级
- 哪些项被转入正式异常任务或升级跟踪
- 是否存在未确认、未接收或信息不足的高风险告警

### Step 6. 审计回放

任务结束后，平台应能通过：

- `/api/v1/tasks/{task_id}`
- `/api/v1/tasks/{task_id}/evidence`
- `/api/v1/tasks/{task_id}/governance`
- `/api/v1/tasks/{task_id}/audit-events`

回放整条告警聚合与分诊轨迹。

## 9. Agentic Pattern 映射

| 阶段 | Pattern | 原因 |
| --- | --- | --- |
| low-risk alert grouping | `single-agent` | 低风险告警的初步归类和摘要适合单 agent 处理 |
| cross-system evidence assembly | `coordinator` | 需要汇总 SCADA、Andon、MES 和 incident log 上下文 |
| triage reasoning | `ReAct loop` | 需要围绕告警时间线、历史异常和业务影响形成分诊建议 |
| downstream escalation | `deterministic workflow + custom business logic` | 正式升级、建任务或后续写入必须由规则控制 |
| safety / quality escalation | `human-in-the-loop` | 任何安全、停线或质量风险都不能由模型直接裁决 |

## 10. 证据与审计要求

必须记录：

- 任务发起人
- 告警时间窗口
- 每次 connector 读取
- 被归并告警的来源引用
- 每条分诊建议的来源说明
- 人工确认说明或升级说明

候选证据源包括：

- SCADA / Andon 告警记录
- MES 设备与生产上下文
- incident log 原始备注
- CMMS 历史异常记录
- 人工补充说明

## 11. 明确禁止的动作

当前和后续试运行阶段都应明确禁止：

- 自动停线或恢复产线
- 自动消除、屏蔽或确认关键告警
- 自动替安全负责人完成安全裁决
- 自动创建真实高风险停线或安全处置动作
- 自动修改 SCADA / Andon / MES 中的真实业务或控制记录

## 12. 回退策略

若 workflow 在任何节点出现异常，应回退到人工告警处置流程：

1. 记录失败原因与当前已汇总证据
2. 保留任务、审计和分诊草稿
3. 由现场负责人手动完成告警筛选和升级
4. 必要时创建后续异常任务补齐跟踪

## 13. 与当前平台能力对齐

当前已支持的通用能力：

- 任务 intake
- 任务查询
- evidence snapshot
- governance 输出
- 审批与修订重提
- 执行开始 / 完成 / 失败
- 审计过滤与任务回放
- 文件 / SQLite / 内存三种本地运行模式

当前尚未支持但该场景后续必须补齐：

- `SCADA / Andon / incident log` connector baseline
- 事件驱动任务输入，而不只是人工 intake
- 告警簇、告警优先级和分诊标签的正式对象模型
- 时间线型 evidence 表达
- 告警去重 / 聚类 / 抑制策略的结构化表达

## 14. 试运行验收标准

进入后续阶段前，至少满足：

1. 能完成 1 次端到端告警分诊任务演示。
2. 能清楚说明哪些告警被归并、哪些被提级、哪些应升级为正式任务。
3. 所有关键读取和分诊节点具备审计记录。
4. 现场角色能看懂分诊建议、责任边界和升级路径。
5. 不发生越权自动停线、自动消警或自动安全裁决。
