# FA v0.2.0 Release Notes

发布日期：2026-03-13

## 1. 发布概览

`v0.2.0` 是 FA 的第一版面向制造协同闭环的可发布版本。这个版本不再只是平台骨架，而是把生命周期、治理、证据、持久化、受限环境验证路径，以及三类高频运营读模型真正接到了统一 API 上。

本次发布覆盖：

- 可审计任务生命周期主链：`intake -> approve -> execute -> complete / fail`
- workflow governance 输出与按 `required_role` 的审批约束
- 内存、文件、SQLite 三种本地运行与回读模式
- 受限环境可执行的 `sandbox-safe` smoke 路径
- 高制造频次协同对象的最小可用读层：`follow-up`、`handoff receipt`、`alert cluster`

## 2. 关键新增能力

### 2.1 核心平台基线

- 任务、审批、执行、失败与重提交流程形成最小闭环
- `MES` / `CMMS` mock 只读 connector 已接入 orchestration context
- evidence snapshot、task governance、task audit replay 已具备 API 与持久化回读能力
- 默认本地端口改为 `8000`，并提供项目内 `sandbox/` 目录作为受限环境工件根目录

### 2.2 高频运营协同读层

- `GET /api/v1/follow-up-items` 提供跨任务 owner queue
- `GET /api/v1/follow-up-monitoring` 提供 follow-up SLA monitoring 聚合视图
- `GET /api/v1/handoff-receipts` 提供跨班次 receipt queue
- `GET /api/v1/handoff-receipt-monitoring` 提供交接确认 backlog monitoring 聚合视图
- `GET /api/v1/alert-clusters` 提供跨任务 alert cluster queue，并带 linked follow-up 摘要
- `GET /api/v1/alert-cluster-monitoring` 提供 cluster backlog、linked follow-up coverage 与 worst-SLA 聚合视图

### 2.3 制造场景建模推进

- `shift handoff` 现在可受控生成 seeded `follow_up_item` 和 `handoff_receipt`
- `alert triage` 现在可受控生成 seeded `follow_up_item` 和 `alert_cluster_draft`
- `alert triage` 已支持更丰富的 `source_system / line_id / triage_label / window` 推断
- `scada` 阈值类 triage 已支持 `sustained_threshold_review` draft 路由

## 3. 运行与升级注意事项

- 默认监听地址为 `FA_SERVER_ADDR=0.0.0.0:8000`
- 本地持久化优先级为 `FA_SQLITE_DB_PATH -> FA_DATA_DIR -> in-memory`
- 需要受限环境验证时，优先使用 `bash scripts/smoke_v0_2_0_sandbox.sh`
- 正式 release gate 仍以 `bash scripts/smoke_v0_2_0.sh` 为准
- 若需要保留本地任务与审计数据，使用独立的 `FA_DATA_DIR` 或 `FA_SQLITE_DB_PATH`

## 4. 已知限制

- 当前 `follow-up`、`handoff receipt`、`alert cluster` monitoring 仍以 repository scan 聚合为主，尚未切换到 dedicated projection
- `MES`、`CMMS`、`QMS` 仍以 mock 或设计基线为主，尚未接入真实企业系统
- 当前版本重点是治理、安全、可追溯和读层闭环，不包含现场自动写回或设备控制动作

## 5. 回退与人工接管

- 若 `v0.2.0` 在试运行环境出现阻断问题，优先回退到 `v0.1.0` 已发布 tag 与对应 artifact
- 回退前先导出或保留当前 `FA_DATA_DIR` / `FA_SQLITE_DB_PATH` 数据，以便审计与问题复盘
- 对高风险或角色不匹配的动作，继续通过 `required_role` 与人工审批链接管，不允许跳过治理边界
- 若 monitoring 视图异常，可先退回 task-scoped `GET /api/v1/tasks/{task_id}`、`/evidence`、`/governance`、`/audit-events` 做人工诊断
