# Governance Controls

## 1. 目的

本文件定义 FA 项目的控制面，确保项目不会因为 0-1 探索而失去边界，尤其是在制造场景下的安全、质量、审批和试运行阶段。

## 2. 范围控制

### 2.1 范围基线

当前范围基线来源于：

- [project-charter.md](/Users/admin/Documents/WORK/ai/fa/docs/project-charter.md)
- [roadmap.md](/Users/admin/Documents/WORK/ai/fa/docs/roadmap.md)
- 当前 milestone 承诺项

### 2.2 范围控制规则

- 新需求默认进入 backlog，不自动进入当前里程碑。
- 未经过评估的想法，不得直接插队进入开发。
- 涉及关键架构、设备、安全、审批的新增范围，必须经过设计和变更评审。

### 2.3 范围冻结点

在以下时间点应冻结范围：

- 版本进入 `Ready for Release`
- UAT 开始前
- 试运行窗口开始前

冻结后仅允许进入：

- Sev-1 / Sev-2 缺陷
- 已批准的紧急变更

## 3. 进度与计划控制

### 3.1 计划层级

| 层级 | 周期 | 负责人 |
| --- | --- | --- |
| Charter / milestone | 月度到季度 | Sponsor / Delivery Lead |
| Roadmap / version | 双周到月度 | PO / Delivery Lead |
| Weekly commitment | 周 | Delivery Lead / Eng Lead |
| Daily task execution | 日 | Eng Lead / owner |

### 3.2 计划更新规则

- 路线图按周滚动更新。
- 里程碑偏差超过 10% 或 1 周时必须升级讨论。
- 单项任务阻塞超过 24 小时必须升级。

### 3.3 进度可视化指标

- 版本 burn-up
- 本周承诺完成率
- Blocked item 数量
- 平均 cycle time
- 缺陷打开 / 关闭比

## 4. 质量管理

### 4.1 质量目标

- 主分支保持可构建、可测试。
- 新增能力必须附带相应测试证据。
- 高风险流程必须具备审批与审计策略。

### 4.2 测试层

| 层级 | 目标 |
| --- | --- |
| Unit test | 保护领域模型和编排逻辑 |
| Integration test | 保护 API、connector、状态机行为 |
| Contract test | 保护外部系统接口契约 |
| UAT | 保护业务流程与试运行场景 |
| Pilot run verification | 保护真实环境中的稳定性和可回退性 |

### 4.3 质量门禁

任何进入发布候选的变更至少满足：

1. `cargo fmt --all`
2. `cargo clippy --workspace --all-targets -- -D warnings`
3. `cargo test --workspace`
4. 关键变更的测试证据已附在 PR 或 issue
5. 文档与变更说明已更新

## 5. 风险、问题、依赖与决策管理

### 5.1 RAID 定义

| 类别 | 说明 |
| --- | --- |
| Risk | 尚未发生，但可能影响目标 |
| Assumption | 当前计划依赖的前提 |
| Issue | 已发生并影响执行的问题 |
| Dependency | 依赖外部资源、接口、决策或环境 |

### 5.2 RAID 管理规则

- 每周 RAID review 更新一次。
- 高风险项必须有 owner、触发条件、应对策略和下次检查日期。
- 已触发风险立即转为 issue，并更新影响评估。

### 5.3 决策管理

- 关键技术决策进入 ADR。
- 关键范围或资源决策进入会议纪要并回写到对应 issue。
- 决策必须包含背景、备选项、结论和影响。

## 6. 变更控制

### 6.1 Change Control Board

CCB 建议成员：

- Delivery Lead / PMP
- Product Owner
- Solution Architect
- Engineering Lead
- QA Lead
- 涉及试运行时增加 Pilot Plant Owner

### 6.2 必须进入 CCB 的变更

- 调整里程碑日期
- 变更试运行目标或现场范围
- 引入新的关键外部系统或供应商依赖
- 改变安全、审批、数据保留、审计边界
- 调整发布节奏或回退策略

### 6.3 变更评估要素

- 对范围的影响
- 对计划和资源的影响
- 对架构和技术债的影响
- 对质量、风险和试运行的影响
- 是否存在更低成本替代方案

## 7. 配置与发布管理

### 7.1 配置管理原则

- 代码、文档、配置样例均纳入版本控制。
- 环境差异必须显式记录，不能靠口头传递。
- 对外接口与审批策略变更必须具备版本可追溯性。

### 7.2 发布管理原则

- 发布基于 tag。
- `CHANGELOG.md` 是版本变更说明的基准文件。
- 发布前必须确认回退策略、已知风险和监控关注点。

### 7.3 试运行前额外控制

- 环境清单确认
- 负责人值守安排
- 回退演练或替代预案
- 现场沟通窗口确认

## 8. 需求和缺陷优先级模型

### 8.1 业务优先级

| 等级 | 说明 |
| --- | --- |
| P0 | 直接影响试运行、关键客户承诺或核心业务闭环 |
| P1 | 影响当前里程碑关键价值 |
| P2 | 重要但可延后 |
| P3 | 优化项或观察项 |

### 8.2 优先级评估因素

- 业务价值
- 风险降低
- 技术依赖解锁程度
- 试运行准备价值
- 开发与验证成本

## 9. 状态汇报与管理看板

### 9.1 周报最低内容

- 本周完成
- 下周计划
- Blocker
- 关键风险变化
- 关键指标
- 需要管理层决策的事项

### 9.2 关键指标建议

- 里程碑达成率
- 版本承诺完成率
- 需求 lead time
- PR 审核时长
- 测试通过率
- Escaped defect 数量
- 试运行事件数与恢复时长

## 10. 升级路径

### 10.1 技术升级

- 先由 Eng Lead 组织解决。
- 超过 1 个工作日未解，升级 Architect。
- 影响版本或现场，升级 Delivery Lead。

### 10.2 业务升级

- 先由 PO 澄清。
- 影响范围或优先级，升级 Sponsor。

### 10.3 发布升级

- QA Lead 和 Eng Lead 联合判断是否进入回退或热修。
- 涉及试运行时，Pilot Owner 有权要求暂停变更。

## 11. 审计要求

以下事项必须在仓库或关联系统中留下可追溯记录：

- 需求来源与验收标准
- 架构决策
- 变更批准
- 发布版本与变更说明
- 高风险缺陷处理
- 试运行 Go / No-Go 决策

## 12. 当前阶段的最小可行治理

为避免 0-1 项目被流程拖死，当前只强制以下最小集：

1. 所有工作都有 issue。
2. 所有变更走 PR。
3. 所有发布有 changelog 和 tag。
4. 所有关键设计有 ADR 或架构文档记录。
5. 每周一次 RAID review 和计划回顾。
6. 任何高风险能力都必须有审批、回退和责任说明。
