# Follow-up and SLA Stage Notes - 2026-03-12

## 1. 为什么 follow-up / SLA 现在必须单独拿出来

很多平台会一直写 workflow spec，却迟迟不补中间那层通用协同对象。结果就是每条 workflow 看起来都能跑，但真正落到日常使用时，owner、时限、阻塞和升级还是散落在文本里。

如果这层不补，平台就很难从“能给建议”走到“能支撑闭环”。

## 2. 这一阶段的创新点

这一阶段的关键，是把 high-frequency workflow 共同暴露出的缺口抽出来，单独变成一份模型对齐清单。

这意味着平台不再只是围绕场景扩写，而开始认真定义：

- 什么是 follow-up item
- owner 在系统里究竟表示什么
- due date 和 SLA 应该怎样区分
- blocked status 与任务主状态之间是什么关系

## 3. 这如何改变世界

制造业里的很多协同损耗，不是因为没人知道该做什么，而是因为没有一层正式对象能把“谁来接、何时完成、卡在什么地方、什么时候升级”稳定表达出来。

这正是 follow-up / SLA 模型真正重要的地方。

## 4. 对自己的要求

- 不把步骤 owner 和业务 owner 混为一谈
- 不在模型未冻结时先堆一批 endpoint
- 不让 SLA 退化成一个孤立的日期字段

## 5. 已经验证的事实

- 当前平台已经有生命周期、evidence、governance 和 audit 骨架
- 当前平台没有一等 follow-up item、owner assignment、due window、blocked status、sla policy
- `班次交接` 和 `告警分诊` 两条 spec 都已经证明这不是局部问题，而是平台问题

## 6. 这次做对了什么

这次做对的地方，是没有继续写第三条、第四条场景 spec，而是先去补它们共同依赖的模型层。

否则再多写几条 workflow，也只会继续把真正的问题藏在自然语言里。

## 7. 这一步如何真正产生影响

这份 alignment checklist 的真正价值，在于它把平台下一步从“继续讲场景”推进到“开始收敛对象模型”。

这会直接影响后续路线：

- read model 和 query 方向会更清楚
- 高频功能会开始共享正式对象，而不是共享口号
- 后续质量、异常、交接、告警场景会更容易落到同一条平台主线上
