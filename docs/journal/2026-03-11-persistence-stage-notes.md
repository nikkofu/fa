# Persistence Stage Notes - 2026-03-11

## 1. 这一阶段为什么重要

真正进入企业现场的系统，不能把“重启就失忆”当成默认前提。

到目前为止，FA 已经具备了任务闭环、修订闭环和治理骨架，但如果状态只停留在内存，所有这些闭环都还是偏演示属性，而不是运行属性。

## 2. 当前阶段的创新点

这一阶段的创新不在于更强的推理，而在于把受治理的 Agent 平台往“可持续运行”推进一步。

这里要证明的是：

- Agent 平台的状态不能只存在进程内存里
- 审计不能只是调试时看的临时输出
- 即使还没进入数据库阶段，也要先建立清晰的持久化边界和运行方式

## 3. 这如何改变世界

工业系统真正改变世界，不是因为它一开始就拥有最大规模，而是因为它能逐步从原型走向可靠运行。

本地文件型持久化看起来不宏大，但它在工程上迈出了一步关键转折：平台开始承认“历史”本身就是一等资产。只要任务和审计能在重启后继续被看到，组织就不再是在和一个短暂会话式工具协作，而是在和一个有连续记忆的系统协作。

## 4. 对自己的要求

- 不把本地文件持久化包装成超出事实的能力
- 保持内存模式和文件模式都清晰可控
- 必须做真实重启后的 smoke test，不只停在单元测试

## 5. 当前阶段结果

这一阶段已经把任务和审计从纯内存推进到了可选的本地文件持久化模式。现在只要设置 `FA_DATA_DIR`，服务就会把任务写入 `tasks/*.json`，把审计事件写入 `audit-events.jsonl`。

更关键的是，这次不是只证明“能写文件”，而是证明“重启后还能继续记得”。我们已经在真实服务进程下完成写入、停止、重启和回读，确认任务与审计在服务重启后仍然可见。

这一步的意义，在于平台第一次拥有了最基础的连续记忆。它当然还不是企业级数据库，也没有事务、锁和复杂查询能力，但它已经跨过了“重启即失忆”的门槛。这是让 Agent 平台真正靠近试运行和真实生产准备的必要一步。
