# Founder Notes - 2026-03-11

## 1. 当前阶段的工作心得

这个项目最重要的认知，不是“怎么把更多 AI 塞进工厂”，而是“怎么让 AI 真正进入工厂工作，又不破坏责任、审批、安全和 SOP”。

传统做法里常见两个极端：

- 只有业务系统，没有智能协同，导致异常处理效率低。
- 只有 AI 对话，没有流程治理，导致无法进入真实生产环境。

FA 的工作方法，是把 AI 从“回答问题的工具”改造成“受治理的协同节点”。这要求我们在一开始就把生命周期、审批、审计、connector 边界和责任归属建起来，而不是等原型写完再补。

## 2. 当前阶段的创新点

当前阶段的创新，不在于模型多强，而在于系统设计方式发生了变化：

### 2.1 把 Agent 能力嵌入制造工作流，而不是嵌入聊天窗口

我们没有把项目做成一个泛化问答助手，而是直接把任务、审批、执行、审计作为一等对象。

### 2.2 用确定性治理包裹概率性智能

LLM/Agent 可以推理，但不能裸奔执行。任务状态机、审批状态机、connector 边界、audit event，都是在给智能能力加上工业级护栏。

### 2.3 让 AI 的行为具备可追溯性

通过 `correlation_id`、审计事件和任务记录，系统能够解释：

- 谁发起了任务
- 读了哪些上下文
- 为什么要审批
- 谁批准了
- 任务当前走到哪一步

这比“模型答得像不像人”更重要，因为制造现场关心的是责任、结果和可追踪。

## 3. 这件事如何改变世界

“改变世界”如果只停留在口号层，没有意义。这个项目真正可能改变世界的方式，是把 AI 从少数人的效率工具，变成工业系统里可治理、可复制、可审计的协同基础设施。

如果这件事做成，影响会体现在三个层面：

### 3.1 改变制造企业处理日常异常的方式

很多制造企业的异常处置仍然依赖老师傅经验。FA 的目标，是把经验、系统上下文和智能推理，沉淀成可复用的协同流程。

### 3.2 改变 AI 落地方式

今天很多 AI 项目停留在 PoC 或问答层，因为无法承担审批、安全和责任链。FA 试图证明：AI 不必绕过治理，也可以进入真实业务流程。

### 3.3 改变人和系统的关系

未来不是人被 AI 替代，而是人、Agent、企业系统和设备形成新的协作网络。真正的价值，是让复杂组织的协作摩擦下降，让决策和执行更快、更稳、更可解释。

## 4. 下一阶段的自我要求

接下来每进入新的阶段任务前，都要先同步：

- 当前版本状态

- GitHub 仓库同步情况
- 风险和下一步
- 本阶段心得与影响说明

如果连项目状态都无法被准确记录，就不配谈“改变世界”。

## 5. 当天阶段推进补记

今天项目已经从“能 intake 和审批”推进到了“能完整驱动生命周期主链”，包括：

- `get`
- `approve`
- `execute`
- `complete`
- `fail`
- 审计回看

这一步的意义是，平台第一次拥有了可反复验证的任务闭环。它离真实制造系统当然还很远，但已经不再只是概念图，而是一个可以持续迭代的治理型原型。

## 6. 关于这次运行态验证的额外心得

今天在 `127.0.0.1:8000` 做 smoke test 时，一个很小但很真实的细节暴露出来了：如果调用方把 `priority` 写成不受支持的值，服务会直接拒绝请求。

这件事看上去不起眼，但它很关键。工业系统不是“差不多能理解就行”，而是必须把边界说清楚、守住、验证掉。真正可落地的 Agent 平台，不能只靠模型理解上下文，还必须靠明确的数据契约、状态机和错误反馈把系统收紧。

换句话说，我们不是在堆功能，而是在把“智能协同”从一种模糊能力，压缩成可以被制造企业接纳的工程产品。
