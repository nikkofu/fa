# Revision Stage Notes - 2026-03-11

## 1. 这一阶段为什么重要

如果一个系统只能表达“提交然后通过”，它更像演示，不像组织真实使用的工作平台。

真实制造企业里，大量任务不会一次获批。它们会被驳回、补充材料、调整方案、再次提交。没有这个过程，所谓的人机协同只是表面闭环。

## 2. 当前阶段的创新点

这一阶段的创新，不是增加一个按钮，而是承认组织协作里“修订”本身就是一等流程。

我们要补的是：

- 驳回不是终点，而是治理内的反馈
- 修订不是脱离系统私下处理，而是受追踪的再提交
- Agent 平台必须支持这种多轮协同，而不是只支持线性 happy path

## 3. 这如何改变世界

如果前面几个阶段在回答“AI 能不能进入工作流”，这一阶段回答的是“AI 进入后，能不能承受真实组织里的不同意见和反复修订”。

真正能改变世界的系统，不是只在顺风局里好看，而是在被质疑、被驳回、被要求重做时，仍然能保持秩序、透明和追踪能力。

## 4. 对自己的要求

- 这次实现必须是闭环，不是只补一个没有验证的 endpoint
- 要把 rejection / resubmit 的状态流和审计流都打通
- 完成后继续同步版本、GitHub、进展和心得

## 5. 当前阶段结果

这一阶段已经把“驳回后修订并重提”从概念补成了真实接口能力。现在任务在被驳回后，可以明确回到 `planned`，再通过 `resubmit` 重新申请审批，而不是只能靠重建任务绕过去。

更重要的是，这次不仅补了 endpoint，还补了服务层测试，验证了两件关键事实：

- 驳回后的任务不能直接执行
- 修订后可以重新进入 `awaiting_approval`

这让平台第一次拥有了更接近真实组织协作的多轮治理能力，而不是只会处理一次性 happy path。

而且这条能力不是停留在单测里。我们已经在 `127.0.0.1:8000` 的真实服务进程下跑过 `intake -> reject -> resubmit -> approve`，证明这条闭环在 API 层也成立。这才是工程上真正有价值的“可用”。
