# Release Readiness Stage Notes - 2026-03-12

## 1. 为什么现在必须做发布准备

0-1 项目最常见的问题，不是做不出功能，而是做出了很多功能，却没有把这些功能收敛成一个可验证、可发布、可交接的版本。

## 2. 这一阶段的创新点

这一阶段的创新，在于把“验证”做成仓库资产，而不是依赖记忆和口头操作。

这意味着：

- smoke test 变成脚本
- 发布门禁变成清单
- 手工验证结果变成文档
- CI 不再只验证编译和单测，也验证主 workflow 的真实 HTTP 行为

## 3. 这如何改变世界

真正能改变行业的软件，不是偶尔跑通一次，而是每次都能被稳定验证、稳定发布、稳定回退。发布准备看起来不炫，但它决定项目能不能从“原型成功”走向“持续交付成功”。

## 4. 对自己的要求

- 把验证流程写成团队可复用的标准资产
- 保持 smoke path 足够接近真实 workflow
- 不把“发布准备”做成空表格，而要有真实可执行脚本和结果

## 5. 已经验证的事实

- 主 workflow 已经可以通过单条脚本完成真实 HTTP 验证
- 文件模式下的重启回读已经进入 smoke 主链，而不是只停留在历史阶段文档里
- CI 和 release workflow 开始共享同一条 smoke gate

这说明项目的验证能力开始从“手工演示能力”转向“可重复交付能力”。

## 6. 这次做对了什么

这次做对的地方，是没有把 QA 和 release readiness 继续留在文档层，而是把它们压进脚本、Makefile 和 workflow。

这样一来：

- 本地验证路径更统一
- 团队协作时不会靠口头传递步骤
- 发布门禁开始具备真实执行力

## 7. 这一步如何真正产生影响

一个想进入制造现场的系统，必须做到“今天能跑通，明天还能按同样方式再跑通”。发布准备资产化，就是把平台从“靠工程师记忆维持”推进到“靠系统化流程维持”。
