# Quality Workflow Stage Notes - 2026-03-12

## 1. 为什么现在要写质量场景 spec

制造业里很多平台会一直围绕设备异常做文章，因为它更容易讲清楚、风险也相对更可控。但如果平台永远不进入质量场景，就很难证明自己能处理真正更靠近组织责任核心的问题。

## 2. 这一阶段的创新点

这一阶段的关键，不是新增一个“质量”关键词，而是把质量场景从候选列表推进成正式设计输入。

这意味着：

- 平台开始面对更高治理密度的场景
- connector、evidence 和审批模型的下一步会更清楚
- 场景扩展不再只是口头上的“以后也可以做质量”

## 3. 这如何改变世界

制造企业是否接受 Agentic AI，往往取决于它能不能处理那些真正需要责任和证据的场景。质量偏差协同就是这样一类场景。把它规格化，说明平台开始认真面对“高价值且高敏感”的业务问题。

## 4. 对自己的要求

- 不把质量场景包装成已经实现的能力
- 明确写出禁止动作，尤其是自动放行和自动报废
- 把当前已支持与尚未支持的能力清楚分开

## 5. 已经验证的事实

- 平台已经具备质量场景所需的一部分通用骨架：task lifecycle、evidence、governance、approval、audit
- 质量场景真正缺的不是“再来一个任务接口”，而是 QMS 相关 connector、evidence 和角色策略

这说明质量场景现在进入 spec 阶段是合适的，但直接进入实现还需要更谨慎的准备。

## 6. 这次做对了什么

这次做对的地方，是没有急着说“下一步就接 QMS”，而是先把质量场景真正拆开。

这样带来的直接价值是：

- 连接器建设更有目标
- 质量治理边界更早暴露
- 后续讨论不会把“质量建议”误写成“质量裁决”

## 7. 这一步如何真正产生影响

一个能进入制造企业的 Agentic AI 平台，最终一定要面对质量场景。但能不能被接受，不取决于它多快接上 QMS，而取决于它是否先把责任、证据和禁止动作讲清楚。这次的 spec baseline，就是在做这件事。
