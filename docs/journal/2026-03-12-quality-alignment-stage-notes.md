# Quality Alignment Stage Notes - 2026-03-12

## 1. 为什么质量场景还需要一层对齐清单

很多团队在写完 workflow spec 之后，会误以为“方向已经清楚了”，于是直接开始接系统。但质量场景不是这样。它真正决定能不能落地的，不是标题写得多完整，而是 connector、证据、审批和 API 是否能对上。

## 2. 这一阶段的创新点

这一阶段的关键，是把“质量很重要”这种方向性判断，继续压缩成实现顺序。

这次最有价值的不是又写了一份质量文档，而是明确了：

- 哪些系统必须先接，哪些可以后接
- 当前 evidence 模型到底能不能承接质量语义
- 为什么 `Quality Manager` 不是一个可以无限后置的问题

## 3. 这如何改变世界

制造业里真正高价值的 AI 平台，最终都要面对质量问题。但质量问题不能靠“先把模型接进去再说”解决。只有把证据、责任和审批角色先对齐，平台才有机会从 demo 走向受控试运行。

## 4. 对自己的要求

- 不把 quality enum 的存在包装成质量能力已经可用
- 不把通用高风险审批路径误写成质量治理已经完成
- 不用含糊的“未来可扩展”掩盖当前确实缺失的结构化质量对象

## 5. 已经验证的事实

- 当前平台已经有质量场景可复用的骨架：task lifecycle、evidence、governance、approval、audit
- 当前平台也已经暴露出明确缺口：没有默认 `QMS` connector，没有 `Quality Manager` 审批策略，没有结构化 impact scope / containment draft
- 当前 API 在 Phase A 足够复用，不需要立刻再发明一批质量专属端点

## 6. 这次做对了什么

这次做对的地方，是没有把“下一步去接 QMS”当作唯一答案，而是先把接入前提讲清楚。

否则就会出现一种常见假进展：系统接上了，但审批角色不对，证据字段也不对，最后根本没法试运行。

## 7. 这一步如何真正产生影响

这份 alignment checklist 的真正作用，是让质量场景第一次具备了工程顺序。

从现在开始，团队讨论的不再只是“质量 workflow 很值得做”，而是更具体的三个问题：

- 什么时候补 mock `QMS`
- 什么时候把 `Quality Manager` 放进正式审批策略
- 什么时候把批次影响和处置建议变成结构化输出

这才是制造业 Agentic AI 从概念走向产品的关键一步。
