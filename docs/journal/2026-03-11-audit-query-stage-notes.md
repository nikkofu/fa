# Audit Query Stage Notes - 2026-03-11

## 1. 这一阶段为什么重要

企业系统的记忆，只有在能被准确查询和复盘时才有价值。

如果审计事件只能全量导出，那么它仍然更像开发调试数据，而不是运行管理工具。真实组织需要的是：某个任务发生了什么、哪次链路触发了什么、为什么走到了当前状态。

## 2. 当前阶段的创新点

这一阶段的创新，不在于把审计做得更“多”，而在于把审计做成可运营的查询能力。

我们要证明的是：

- 治理型 Agent 平台不仅能记录事件，还能按业务语义回放事件
- `task_id` 和 `correlation_id` 不只是日志字段，而是运行管理主键
- 审计能力要从“存证”进化到“定位、解释、复盘”

## 3. 这如何改变世界

真正能进入工业现场的 AI 平台，不能只会做动作，还必须能解释动作。未来企业信任 AI，不是因为它永远不出错，而是因为它出问题时仍然能被追踪、被定位、被复盘。

审计回放能力看起来很底层，但它决定了系统是否能进入更高信任等级的组织运行环境。

## 4. 对自己的要求

- 这次要做出真正可用的回放入口，而不是只补一个低价值过滤参数
- 保持内存模式与文件模式一致
- 完成后继续同步版本、进展、GitHub 基线和工作心得

## 5. 当前阶段结果

这一阶段已经把审计从“能保存”推进到了“能回放、能定位、能按业务主键查询”。现在我们既能用 `/api/v1/audit/events` 按 `task_id / correlation_id / kind / approval_id` 过滤，也能直接用 `/api/v1/tasks/{task_id}/audit-events` 回看单任务的完整历史。

更重要的是，这次验证不是停在单元测试里。我们已经在 `FA_DATA_DIR` 文件模式下跑通真实 HTTP 验证，确认按链路和按任务的查询都可用。这意味着平台的审计能力开始具备运维、排障和试运行复盘价值，而不再只是开发调试数据。
