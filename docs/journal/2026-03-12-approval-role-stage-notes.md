# Approval Role Enforcement Stage Notes - 2026-03-12

## 1. 为什么这一步重要

制造场景里最危险的不是系统没有审批，而是系统看起来有审批，实际上任何人都能代批。那样的流程会制造一种虚假的安全感。

## 2. 这一阶段的创新点

这一阶段的关键，不是再加一个新的治理字段，而是让治理字段开始真正约束行为。

这意味着：

- `required_role` 不再只是响应里的说明
- 错误角色会被系统直接拒绝
- API 行为开始与组织责任边界对齐

## 3. 这如何改变世界

Agentic AI 真正进入工厂，不只是因为它会规划，还因为它知道哪些决定自己不能放给错误的人做。只有把责任边界执行起来，AI 才更像被组织接受的协同系统，而不是一个会生成建议的外围工具。

## 4. 对自己的要求

- 把角色约束放在领域层，而不是只靠接口层拦截
- 不因为加约束而破坏主 workflow
- 文档、脚本、示例必须和真实治理规则保持一致

## 5. 已经验证的事实

- 高风险任务的审批角色现在会被强校验
- 不匹配角色的审批请求会返回 `403`
- 匹配角色的主 workflow、测试和真实 smoke 仍然成立

这说明平台开始能把“谁有权批准”从说明文字推进成运行时约束。

## 6. 这次做对了什么

这次做对的地方，是没有把角色校验写成一段只服务当前 HTTP 入口的判断，而是压进 `ApprovalRecord` 自己的生命周期里。

这样带来的直接价值是：

- 治理规则更一致
- 持久化模式不会出现行为分叉
- 后续如果增加 CLI、批处理或别的入口，也会自动继承这层约束

## 7. 这一步如何真正产生影响

当平台能明确拒绝错误审批角色时，组织才会更愿意把高风险异常处理交给系统协同。因为这不再只是“系统建议由谁审批”，而是“系统真的守住了谁能审批”。
