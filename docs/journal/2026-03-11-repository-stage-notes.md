# Repository Stage Notes - 2026-03-11

## 1. 这一阶段为什么重要

很多 AI 原型失败，不是因为第一次 demo 做不出来，而是因为状态、责任和外部依赖都被写死在第一版代码里，导致第二步根本走不出去。

当前 FA 已经拥有生命周期闭环，但任务状态仍停留在编排器内部的内存结构里。这意味着：

- 系统可以演示，却还不够可替换
- 系统可以跑通，却还不够可演进
- 系统可以表达流程，却还没有为真实持久化留出稳定边界

## 2. 当前阶段的创新点

这一阶段的创新，不是增加更多“智能”，而是给智能协同系统建立更像工业软件的骨架。

我们要做的事，本质上是在证明：

- Agent 系统也需要清晰的 repository boundary
- 编排逻辑不应该直接依赖某一种状态存储方式
- 只有把状态管理抽出来，后续审计、回放、重试和持久化才有工程落点

## 3. 这如何改变世界

如果说前一阶段证明了“AI 可以进入制造任务闭环”，这一阶段要证明的是“这种闭环不是一次性 demo，而是可以继续长成企业级系统”。

真正改变世界的，不是一个会说话的模型，而是一套能持续被组织信任、扩展、审计和迁移的协同基础设施。repository abstraction 看起来很底层，但它决定了这个项目未来能不能承接真实工厂的数据、责任和运行历史。

## 4. 对自己的要求

这一阶段必须守住三个原则：

- 抽象必须服务后续持久化，而不是为了抽象而抽象
- 行为必须保持稳定，不能为了重构破坏现有生命周期
- 每完成一个阶段切片，都继续同步版本、文档、GitHub 基线和心得

## 5. 当前阶段结果

这一阶段已经把任务状态存取从编排器内部拆出来，形成了显式的 `TaskRepository` 边界，并保留了当前生命周期 API 与测试行为。

这一步的意义，是让平台第一次具备“换存储而不改主编排逻辑”的结构条件。它还没有解决真实持久化，但已经把未来真实持久化需要踩的路修出来了。这类工作表面上没有新页面、没有更炫的 AI 展示，但它决定了系统能不能继续长大，能不能真正走进企业。

更重要的是，这次不是只停在单元测试层。我们已经在 `127.0.0.1:8000` 下把真实 HTTP 进程跑起来，验证了 `intake -> approve -> execute -> complete -> get` 仍然成立。这说明抽象不是纸面重构，而是保持了运行态闭环。
