# Sandbox-Safe Smoke Stage Notes - 2026-03-12

## 1. 为什么这一步重要

很多工程团队会默认“能在自己电脑上起服务就算可验证”，但一进沙箱或受限 runner，端口监听权限就可能直接把验证链路打断。

## 2. 这一阶段的创新点

这一步的关键不是换一个端口，而是换一个更稳健的验证思路。

这意味着：

- 受限环境不再依赖本地 TCP 监听
- 仍然保留 HTTP 路由级别的行为验证
- 文件持久化和重启回读依旧能被真实覆盖

## 3. 这如何改变世界

真正能持续交付的系统，不能只在“权限足够大”的环境里验证。把 smoke 路径做成既有真实 listener 版，又有 sandbox-safe 版，项目的工程韧性会明显更强。

## 4. 对自己的要求

- 不把 sandbox-safe 路径包装成真实网络验证
- 保持它与正式 workflow 语义一致
- 临时数据必须被明确收拢到项目内可控目录

## 5. 已经验证的事实

- 进程内 HTTP 路由验证可以覆盖主 workflow
- 文件模式重启回读可以在 `sandbox/` 下完成
- 审批角色强校验在受限环境路径中同样成立

这说明项目已经不再被单一运行环境绑死。

## 6. 这次做对了什么

这次做对的地方，是没有去和沙箱权限硬碰硬，而是承认监听端口可能不可行，然后把验证能力重新设计成更可迁移的形式。

这样带来的直接价值是：

- 本地受限环境也能稳定跑 smoke
- 自动化环境更容易复用同一路径
- `sandbox/` 把临时工件集中在仓库可控边界内

## 7. 这一步如何真正产生影响

一个想进入真实企业环境的系统，必须适应各种受控执行条件。sandbox-safe smoke 不是“降级版验证”，而是把验证能力做得更稳、更广、更适应现实环境限制。
