# Alert Triage Stage Notes - 2026-03-12

## 1. 为什么告警分诊必须尽快规格化

制造现场最不缺的往往不是信号，而是信号之后的判断。告警太多、太碎、太重复时，真正稀缺的是一条稳定的分诊路径，而不是再多一块显示屏。

如果平台不能处理告警流，它就很难进入制造现场最密集、最实时的协同节奏。

## 2. 这一阶段的创新点

这一阶段的关键，不是写一个“告警总结器”，而是把告警流写成正式 workflow。

这意味着平台开始明确三个真正重要的问题：

- 哪些告警应该被归并，哪些不该
- 哪些告警只做提醒，哪些必须升级
- 哪些动作只能由人工裁决，系统绝不能越过

## 3. 这如何改变世界

很多制造系统其实已经能发出告警，但仍然无法形成高质量协同。问题不在于没有信号，而在于没有一层能把信号变成可理解、可分级、可路由对象的操作层。

这正是 Agentic AI 在制造语境里更有价值的位置。

## 4. 对自己的要求

- 不把告警分诊包装成自动化控制
- 不让“事件驱动”成为放松治理边界的借口
- 不提前承诺当前没有的实时连接器和告警对象模型

## 5. 已经验证的事实

- 当前平台已经具备承接告警分诊的一部分基础：task、evidence、governance、audit
- 当前平台也存在明确缺口：没有 `SCADA / Andon / incident log` connector，没有事件驱动入口，没有时间线型 evidence
- 告警场景真正缺的不是新 endpoint，而是事件对象和分诊边界

## 6. 这次做对了什么

这次做对的地方，是没有把告警场景写成“更快地总结告警”，而是把它写成“聚合 + 去重 + 分级 + 路由 + 升级边界”的完整协同 workflow。

这样后续如果继续做 follow-up / SLA，对象模型就不会漂浮在空中，而会落在真实的高频输入上。

## 7. 这一步如何真正产生影响

这份 spec baseline 的真正价值，在于它让 FA 第一次清楚定义了“如何面对实时事件流而不越过治理边界”。

这会直接影响后续路线：

- 事件到任务的受控转换会变得更重要
- timeline evidence 会从“可选增强”变成必要能力
- 高时效协同会成为平台主线的一部分，而不只是附属场景
